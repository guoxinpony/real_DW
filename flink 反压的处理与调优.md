### 1.  什么是反压？

反压就是**下游处理速度跟不上上游发送速度**，导致数据在缓冲区堆积，压力向上游传导的现象



### 2. 如何定位反压？

1.  Flink Web UI
   - [ ] 如果一个算子 backPressured 很高，说明下游是瓶颈
   - [ ] 如果一个算子 **busy 很高**而 backPressured 为 0，说明当前算子就是瓶颈点
2. Checkpoint 时长：反压会直接导致 **Checkpoint 变慢甚至失败**。因为在反压严重时，Checkpoint Barrier 无法在算子之间快速流动（Barrier 对齐需要时间）



### 3. 反压的原因和解决方案

##### 1. 数据倾斜

由于 keyBy 分组不均匀，导致某一个 Subtask 处理的数据量远大于其他 Subtask：

- [ ] 处理记录数（Records Received）： 如果某**几个 Subtask 处理的数据量远大于其他 Subtask**（例如：有的处理了 1 亿条，有的只处理了 10 万条）
- [ ] **堆内存占用（Heap Memory）：** 倾斜的 Subtask 内存占用会极高
- [ ] **Busy Time：** 倾斜的 Subtask 长期处于 100% 繁忙状态，而其他 Subtask 处于空闲



**解决方案：**

1. **两阶段聚合：局部聚合 + 全局聚合**
   - [ ] 某个特定的 Key数据量极大，给每个 Key 拼接一个随机后缀，让原本一个 Key 变成多个 Key
   - [ ] 局部聚合
   - [ ]  去掉后缀，进行全局聚合
2. **预聚合**
   - [ ] 在 Flink SQL 中进行 **COUNT DISTINCT** 或 **SUM** 操作时，开启 Flink SQL 的微批处理优化
   - [ ] 让数据先在算子本地进行预聚合，减少发往下游的数据量
3. **处理空值**
   - [ ] 日志数据中，某些字段缺失导致**大量的 null 聚合到了同一个 Task**
   - [ ] 如果 null 数据不需要，直接 过滤掉
   - [ ] ；如果需要保留，可以给 null 值**分配随机字符串**，将压力分散



##### 2. Sink 端反压

如果外部数据库写入慢（例如 MySQL 锁等待、HBase RegionServer 压力大），Sink 算子就会被阻塞，压力迅速回传



**解决方案：异步 I/O**

**有序模式 (Ordered)**

- **特点：** 即使后面的数据先返回了，也要等前面的数据返回后，才能一起按下游要求的顺序输出。
- **代价：** 会增加一定的延迟（Latency），因为需要缓存先返回的数据来保证顺序。
- **适用：** 对数据顺序有严格要求的业务场景。

**无序模式 (Unordered)**

- **特点：** 谁先返回谁就先输出。
- **优点：** 性能最高，延迟最低。
- **注意：** 仅适用于对顺序不敏感的场景（例如只是一次简单的维度补全）。
- **特殊情况：** 如果在无序模式下使用了 **Watermark**，Flink 会以 Watermark 为界限，Watermark 之前的数据仍然不能跑到 Watermark 之后去，从而在性能和正确性之间取得平衡。



**使用异步 I/O 的注意事项**

1. **超时处理 (Timeout)：** 必须设置合理的超时时间，否则某个请求卡死会导致整个流停滞。
2. **并发压力 (Capacity)：** 异步请求不是无限的，需要通过 **Capacity** 参数限制同时在途的请求数，防止压垮外部数据库。
3. **算子状态：** 异步 I/O 的在途请求会保存在状态（State）中，如果设置的**并发量（Capacity）**过大，在 **Checkpoint** 时会增加状态量的大小。
4. **外部驱动支持：** 外部数据库必须提供**异步客户端驱动**（如 Vert.x 提供的 Redis/MySQL 异步驱动）。如果没有异步驱动，只能通过线程池模拟异步，效果会大打折扣。
