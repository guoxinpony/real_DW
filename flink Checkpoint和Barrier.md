### 1. Checkpoint 的基本原理

- [ ] 通过在数据流中插入特殊的控制事件——**Barrier**，来对数据流进行“切分”
- [ ] **JobManager**定期触发 Checkpoint，向所有 Source 节点发送命令
- [ ] **Barrier 注入**：Source 收到命令后，在当前数据流中注入一个带编号的 Barrier
- [ ] **Barrier 传递**：Barrier 随普通数据一起向下游流动，它**不可超越**数据，也不会被数据超越
- [ ] 当一个算子（Operator）接收到**所有**上游发送过来的 **Barrier n** 时，它会触发**“快照”**动作：**将当前的状态异步保存到状态后端**（如 RocksDB），并向 JobManager 发送确认



### 2. Barrier 对齐 精准一次

- [ ] 状态**只包含 Barrier n 之前的数据，不包含 Barrier n 之后的数据**
- [ ] **暂停处理**：算子从输入流 A 收到 Barrier n，但流 B 的 Barrier n 还没到
- [ ] **阻塞流 A**：算子会停止处理来自流 A 的数据，并将后续到达的数据存入 **Input Buffer** 中
- [ ] **等待流 B**：继续处理流 B 的数据，直到流 B 的 Barrier n 到达
- [ ] **快照与广播**：此时，**所有输入流的 Barrier n 到齐**（即“对齐”完成）。算子对 State 做快照，并将 Barrier n 发向下游
- [ ] **释放 Buffer**：快照完成后，算子才开始处理 Input Buffer 中积压的流 A 的数据



### 3. Barrier 对齐 至少一次

- [ ] 为了降低延迟，算子在收到流 A 的 Barrier n 后，**不阻塞**流 A
- [ ] 即使流 B 的 Barrier n 还没到，流 A 属于 Checkpoint n+1 的数据也会被直接处理并**更新到当前 State 中**
- [ ] 等流 B 的 Barrier n 到达后，算子直接做快照
- [ ] 此时的快照里，**包含了流A中“本该属于下一个 Checkpoint”的数据**，如果发生故障重启，流 A 的那部分数据会被 Source 重放，导致被重复计算



| **特性**         | **精准一次 (Exactly-Once)**                                | **至少一次 (At-Least-Once)**     |
| ---------------- | ---------------------------------------------------------- | -------------------------------- |
| **Barrier 行为** | **必须对齐**，会阻塞先到的流                               | **不对齐**，所有流直接处理       |
| **性能影响**     | 在**反压（Backpressure）**严重时，对齐会导致明显的延迟抖动 | 延迟低，吞吐更稳定               |
| **数据一致性**   | 完美一致，无重复                                           | 可能存在重复数据                 |
| **适用场景**     | 金融交易、精确统计                                         | 监控告警、对重叠计算不敏感的场景 |



### 4. 非对齐检查点UC 精准一次

**原理：**

- [ ] Barrier 不再排队等待。一旦 Barrier 到达，算子**立即快照当前状态**，并把**算子输入输出管道中还没处理完的数据（In-flight data）**也一起存入 Checkpoint
- [ ] **Barrier 超越**：当第一个 Barrier n 到达算子时，它不再排队等待，而是直接**跳到**所有输入缓存队列的最前面
- [ ] **立即触发快照**：算子立刻对当前状态进行快照
- [ ] **“在途”数据 In-flight data**：Barrier 跳过了队列中积压的数据，这些数据本该在 Barrier n 之前处理，作为 Checkpoint 的一部分保存到 HDFS中
- [ ] **向下游广播**：Barrier 立即发往下一个算子



**为什么非对齐检查点依然是精准一次，而不是至少一次？**

- [ ] **恢复逻辑**：如果任务挂了，Flink 恢复时**不仅恢复算子的 State**，还会把 Checkpoint 里保存的那些“在途数据”**重新放回**输入和输出缓存中
- [ ] **结果**：对算子来说，它恢复到了“Barrier 到达那一刻”的状态，并且管道里该有的数据一条没多、一条没少。这在逻辑上和“对齐”后的效果完全一致
- [ ] **优点：**解决生产环境中反压（Backpressure）**导致 Checkpoint 超时最有效的手段**



**Flink 2.0 环境下，非对齐检查点的最新变化？**

- [ ] **存算分离 (Disaggregated State - ForSt)**：在 Flink 2.0 中，状态由于是持续异步上传到远端存储的，Checkpint 变得极其轻量（轻量级快照）
- [ ] **对齐切换 (Alignment Timeout)**：先尝试对齐，如果 *10 秒内对齐不了*（说明有反压），再自动降级**切换到非对齐**



| **检查点类型**         | **Barrier 行为** | **是否阻塞算子**         | **是否包含管道数据** | **一致性语义** |
| ---------------------- | ---------------- | ------------------------ | -------------------- | -------------- |
| **精准一次 (Aligned)** | **FIFO 排队**    | 是（对齐期间阻塞先到流） | 否                   | Exactly-once   |
| **至少一次 (Aligned)** | **FIFO 排队**    | 否（不阻塞，直接处理）   | 否                   | At-least-once  |
| **非对齐 (Unaligned)** | **跳跃/超越**    | 否（立即触发）           | **是**               | Exactly-once   |
