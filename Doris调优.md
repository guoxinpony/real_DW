###  Doris查询怎么调优

- [ ] **数据模型选择。** 搜索词频率这类**聚合指标用 Aggregate Key 模型**，Doris 写入时自动预聚合，查询几乎零计算开销；**独立访客 UV 这类需要实时更新的指标用 Unique Key 模型**，天然支持 Flink 撤回流的 Upsert 语义；**明细事实表用 Duplicate Key**，充分发挥列存扫描优势。
- [ ] **分区分桶设计。** 按事件日期做 Range 动态分区，查询带上日期条件就能直接裁剪分区，IO 减少数量级。分桶 Key 选高基数维度列，Bucket 数设为 BE 总核数的倍数，保证并行度最大化。
- [ ] **查询层面的优化。** 对于看板里高频的聚合 SQL，提前**创建物化视图**，查询命中后直接读预计算结果，不需要扫宽表。**多表关联的场景用 Colocation Join**，把关联表放同一个 Colocation Group，Local Join 完全避免 Shuffle。另外**开启查询结果缓存**，监控大盘同一个 SQL 被几百个用户并发刷的场景，缓存命中率能到 90% 以上。所有 SQL 必须让过滤条件命中分区列和 Sort Key，用 EXPLAIN 确认有分区裁剪和索引扫描。
- [ ] **第四是 Compaction 控制，这是实时写入场景最容易忽视的点。** Flink 写入 Doris 用 Stream Load 攒批提交，每批 10 万行或 5 秒触发一次，避免小版本堆积压垮 Compaction，否则 Tablet 版本数爆炸会直接触发 -235 报错导致查询失败。



### Aggregate Key — 聚合模型

- [ ] **核心机制**：表里的列分两类，Key 列和 Value 列。每次写入数据时，如果 Key 相同，Doris 在 Compaction 阶段会自动把 Value 列按照你定义的聚合函数合并，比如 SUM、MAX、MIN、REPLACE 等。
- [ ] 以搜索词频率统计表为例，`keyword + time_window` 作为 Key，`search_count` 设为 SUM 聚合。Flink 每次写入一条"doris 出现了 3 次"，Doris 自动累加，查询时直接读结果，不需要再做 GROUP BY 计算，速度极快。
- [ ] **适合**：预先知道聚合方式、数据量大但维度固定的指标类宽表。
- [ ] **注意点**：Value 列失去了原始明细，查不到单条记录的原始值，只能查聚合后的结果。



### Unique Key — 唯一模型

- [ ] **核心机制**：本质是**特殊的 Aggregate Key**，所有 Value 列的聚合函数都是 **REPLACE**，即相同 Key 写入时，新值直接覆盖旧值，永远保留最新的那条。Doris 2.0 引入了 Merge-on-Write 模式，写入时就做去重，查询不需要再合并，性能大幅提升
- [ ] **例子**：UV 汇总表，`channel + dt` 作为 Key，`uv_count` 是 Value。Flink 的撤回流（UPDATE_BEFORE / UPDATE_AFTER）写进来，Doris 用 Upsert 语义直接把旧 UV 值替换成新的，看板查到的永远是最新结果
- [ ] **适合**：有实时更新需求的指标，比如订单状态、累计 GMV、当日 UV 这类需要频繁刷新的数值。
- [ ] **注意点**：写入频繁时 Compaction 压力比 Aggregate Key 大，需要控制 Flink 的写入批次大小



### 为什么使用Colocation Join比普通JOIN性能更好

- [ ] 在分布式环境下，两张表的数据分布在不同的 BE 节点上。执行 Join 时，Doris 需要保证**相同 Join Key 的数据在同一个节点上才能配对**
- [ ] 如果没有 Colocation，Doris 就必须做一次 **Shuffle**：把两张表的数据都按照 Join Key 重新哈希分发，让相同 Key 的数据汇聚到同一个节点。这个过程涉及大量网络传输，是 Join 性能的最大瓶颈
- [ ] Colocation Join 两张表放进同一个 Colocation Group，Doris 会强制这两张表**分桶数相同、分桶 Key 相同、副本分布相同**
- [ ] Join 执行时，节点 A 只需要读取自己本地的 Bucket 3，**把两张表的数据在本地直接配对，完全不需要跨节点传数据**。 每个节点独立处理自己的 Bucket，互不干扰，这就是 **Local Join**
- [ ] **建表时就要规划好**，Colocation Group 一旦建立，后期改分桶 Key 或分桶数会破坏 Colocation 关系，代价很高
- [ ] **Join Key 必须是分桶 Key**：如果你的 Join 条件是 `ON a.order_id = b.order_id`，但两张表是按 `user_id` 分桶的，Colocation 不会生效，还是会退化成 Shuffle Join



### 在Doris中，Bucket 数该如何确定

- [ ] Bucket 数 ≥ BE 总核数，通常是**核数的整数倍**
- [ ] 如果 Bucket 数 < 核数，比如 8 个 Bucket、32 个核，那最多只有 8 个核在干活，剩下 24 个核全部空闲，**CPU 资源白白浪费**
- [ ] **为什么Bucket 数是BE 总核数的2 倍或 4 倍：**  **数据倾斜的容错**。如果某个 Bucket 的数据特别多，它处理得慢，其他核处理完自己的 Bucket 后可以接着处理下一个，避免整体等最慢的那个核。Bucket 数越多，倾斜的影响越分散。
- [ ] **为什么Bucket 数是BE 总核数的2 倍或 4 倍：**  **未来扩容的弹性**。如果集群从 10 台扩到 20 台，BE 核数翻倍，Bucket 数如果只等于旧核数，扩容后又不够用了。提前设成 2 倍，扩容后直接能用满
- [ ] **项目参数：** 10 台服务器，假设每台 16 核，BE 总核数 160。Bucket 数设为256，bucket 过多会导致 Tablet 数量爆炸，FE 的元数据管理压力会很重
- [ ] 单个 Tablet(Bucket) 的数据量保持在 **1GB ～ 10GB** 之间是比较健康的范围，可以用 `总数据量 ÷ 目标 Tablet 大小` 来反推 Bucket 数，和核数倍数的结果取一个合理的交集



### 分桶为什么用高基数列

- [ ] 分桶的目的是把一个分区内的数据**均匀打散**到多个tablet上，每个tablet是Doris最小的存储和计算单元，可以被分配到不同的BE节点并行处理
- [ ] 分桶用的是Hash策略——对分桶列的值做hash取模，决定这条数据落到哪个bucket
- [ ] 如果你选了一个低基数列（比如性别，只有男/女两个值），hash取模后数据只会落到两个bucket里，其他bucket是空的，导致数据严重倾斜
- [ ] 选高基数列（比如用户ID、订单ID）就是为了让hash之后的分布尽量均匀，数据能够平均分配到所有bucket，查询时每个BE节点处理的数据量接近，不会出现一个节点特别慢拖累整体的情况
- [ ] 如果你的查询条件经常命中分桶列（比如 `WHERE user_id = 123`），Doris可以直接定位到对应的bucket做**桶裁剪**，跳过其他所有bucket



### 物化视图为什么能优化查询性能

- [ ] 物化视图的核心是**预计算**。它把一些高频的聚合查询结果提前算好，物理存储下来
- [ ] 查询命中时直接读预计算结果，不用每次都扫描原始明细数据做实时聚合
- [ ] **底层数据变更时会自动增量更新**物化视图的内容
- [ ] 例如在计算 "每个门店每小时的GMV" 的场景下，如果没有物化视图，每次查询都要扫描DWD层几千万条订单明细做GROUP BY + SUM；建了物化视图之后，这个聚合结果已经预先算好存在那里了，查询直接读取
