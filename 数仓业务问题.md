### 维度建模：为什么实时数仓使用了维度建模，而不是ER模型

- [ ] **第一是查询性能。** ER 建模遵循三范式，数据拆得很散，查询需要多表 Join。但在实时场景下，Join 的代价非常高，维度建模通过提前打宽，查询时一张宽表就能搞定，延迟低很多
- [ ] **第二是 State 可控。** 这是实时场景特有的问题。Flink 里 Join 和聚合都要维护 State，ER 建模多表关联会导致 State 无限膨胀，容易 OOM。而维度建模中维度表数据量小、变化慢，可以直接广播到内存，State 大小可预估，系统稳定性好很多。
- [ ] **第三是业务可读性。** 维度建模把数据分成事实表和维度表，和业务语言天然对齐，业务方可以自助查数，不需要每次都找数据团队。在实时数仓里业务方需要实时盯看板，这一点尤其重要。
- [ ] **需要额外处理缓慢变化维（SCD）** 比如用户的城市变了、商品的类目调整了，维度表的历史数据要怎么处理？  新增一行，加 start_date/end_date + is_current 标记 → 保留历史，Flink 可以通过监听维度表的 CDC 数据来实时维护



### 场景：flink 从kafka读取数据， 如果这时候改变kafka的分区数量，会发生什么

- [ ] 首先看flink是否开启**动态分区发现机制**
- [ ] **未开启动态分区发现**，新分区数据永远不会被读取，直到作业重启
- [ ] **已开启动态分区发现**，Flink 的 `KafkaSourceEnumerator` 会按照你设置的间隔**定期轮询** Kafka broker，检测分区变化，如果轮询发现新分区， 将新分区分配给现有的 SourceReader subtask，每个 subtask 从新分区的 earliest/latest offset 开始消费（取决于配置）
- [ ] **Kafka 不支持直接减少分区数，分区是数据的物理存储边界，减少分区意味着数据丢失，因此被禁止**
- [ ] 如果某个分区的 leader 不可用（broker宕机），Flink 的行为是subtask 尝试 fetch 数据并触发重试，超过重试次数并且TaskManager 抛出异常，Flink 触发 failover，从最近 Checkpoint 恢复

### HBase RowKey 设计思路

- [ ] 写入要**分散**  →  数据均匀分布到各个 RegionServer，避免写入热点
- [ ] 查询要**聚集**  →  相关数据存在一起，减少扫描范围，提高查询效率
- [ ] 针对热点，我会在 RowKey 前面加盐，把 userId 做 hash 取模，均匀打散到各个 Region
- [ ] 针对查询，实时数仓最常见的是查最新数据，所以**时间戳我会做反转，让最新数据排在最前面**，range scan 直接从头扫就行
- [ ] **RowKey 长度要控制**，因为它会存在内存、磁盘和索引里，太长会有存储放大的问题
- [ ] RowKey 结构是 **盐值 + userId hash + 反转时间戳 + 事件类型**，兼顾了写入分散和查询性能



### Hbase和redis如何保障缓存一致性的

- [ ] Checkpoint 记录 Binlog Offset，失败重启后从上一个 Checkpoint 恢复，重新执行，再次尝试写 HBase + 删 Redis
- [ ] **HBase 写入成功，Redis 删除失败** → Checkpoint 未完成 → Flink 重启后重放这条 Binlog → 再次执行写 HBase（幂等，不影响）+ 删 Redis
- [ ] **最终效果**：只要 Flink 任务最终成功处理，HBase 和 Redis 一定都会被正确更新
- [ ] **短暂不一致窗口**：在故障恢复期间，Redis 可能仍有旧数据，但 TTL 到期后也会自然失效兜底



### 如果上游mySQL业务数据库有错误数据，如何修复或者调整，对于已经存在于kafka中的数据如何修复？

- [ ] **如果是维度数据出错：** 上游业务库执行 UPDATE 修正维度数据， Flink CDC 捕获到 Binlog 变更事件，Flink 将修正后的数据写入 HBase 覆盖旧值并删除redis缓存
- [ ] **如果是事务事实表数据出错：** 在只增不改的场景下，业务库插入一条**冲正/退款记录**，Flink CDC 捕获后作为新的事务事件流入 DWD
- [ ] **如果是累积快照事实表出错：** 在同一行会被多次更新的情况下，上游修正后产生的 UPDATE 事件，Flink CDC 会自动捕获。Flink 收到后更新对应的状态和下游输出 
- [ ] **Kafka 中已经存在的错误数据：** 要么等 Kafka 消息过期， 要么通过重置 Flink 作业的 Kafka Consumer Offset，配合从 Savepoint 重启来重新消费和处理。
- [ ] **DWS 层Doris Primary Key 模型：** 上游修正后 Flink 输出的新数据会按主键 Upsert，直接覆盖旧的聚合结果
- [ ] **DWS 层Doris Aggregate Key 模型：** 写入一条**负值补偿记录**，让 SUM 之后的结果抵消掉错误值，然后再写入正确值
- [ ] **兜底方案：离线修补** 从业务库拉取修正后的全量/增量数据，通过离线批处理（比如 Spark 批任务或 Flink 批模式）重新计算受影响时间段的 DWS/ADS 指标，然后写回 Doris 覆盖错误的聚合结果。



### Flink Checkpoint 策略配置

- [ ] env.enableCheckpointing(60000)，**每60秒做一次 Checkpoint**；间隔太短，Checkpoint 频繁，barrier 注入频繁，占用大量 IO 和网络资源，影响正常数据处理，吞吐下降；间隔太长，故障恢复时要重放大量数据，恢复时间长
- [ ] **事实数据（GMV、订单）Exactly-Once**，不能有重复；**日志类数据（PV、UV），At-Least-Once 也可接受**，下游去重
- [ ] **超时时间 3分钟：设为 Checkpoint 间隔的 3 倍**
- [ ] **最小间隔防止 Checkpoint 堆积：30秒**；如果 Checkpoint 做了80秒才完成，而间隔设的是60秒，上一个还没完，下一个已经该触发了，导致Checkpoint 堆积
- [ ] **State Backend 选择RocksDB**：对于基于内存的后端，State 全在 JVM 堆内存，读写快，但受堆大小限制（几百 MB），RocksDBState 存在本地磁盘（HDFS）中，支持超大 State（TB 级别），读写有序列化开销，比内存慢



### 如何定位数据倾斜KEY

- [ ] 发现倾斜信号：有没有倾斜， 看Flink Web UI，看各个 **subtask 的接收记录数**，某个subtask接收数据量远大于其他，证明存在倾斜；**看 反压指标Backpressure Status**，如果**飘红**，说明处理速度远跟不上数据流入，证明存在数据倾斜；在**flink 1.13以后**，直接查看**data skew指标**判断数据倾斜
- [ ] 定位 key：SQL 统计 key 分布，例如统计数据量 Top 20 的 key；
- [ ] 进一步分析：少数 key 倾斜 vs 全局倾斜 ，如果 top10 占 80%+ → 少数 key 倾斜，如果 top10 只占 30%  → 全局倾斜
- [ ] 针对group by还是join, 局部还是全局倾斜做处理



### Redis缓存维度数据用的哪种数据结构？

- [ ] **背景：** 需要缓存hbase中的维表数据，key是用户ID，value是用户的多个维度字段（姓名、等级、地址等），**Flink Lookup Join查询时经常只需要其中几个字段**
- [ ] **为什么不选String存JSON：** 整存整取，每次查询都要GET完整JSON再反序列化，即使你只需要一个字段。更新某个字段时要GET-修改-SET，存在并发覆盖风险，也浪费网络带宽
- [ ] **为什么不选多个String（每个字段一个key）：** 比如 `user:123:name`、`user:123:level`，**key数量会膨胀几倍甚至十几倍，内存开销大**。**多个字段无法原子设置同一个过期时间**，要么逐个设TTL要么用pipeline，管理复杂。逻辑上属于同一个实体的数据被打散了，不利于维护
- [ ] **为什么选hash:**  一个hash key对应一个用户，内部的field就是各维度字段。HGET/HMGET支持按字段粒度查询，Flink只取需要的字段，省带宽；所有字段归在一个key下，设一次TTL（5-6分钟随机）就能整体过期，管理简单；field数量少的时候底层用ziplist编码，内存比等量的多个String key更紧凑。



### Flink CDC OP字段值的含义，以及写入kafka时使用debezium-json格式的op字段值

- [ ] Flink内部用RowKind（+I、-U、+U、-D）来标记变更类型，
- [ ] `+I` Insert，新增记录，对应全量快照阶段读取的存量数据，以及增量阶段捕获的INSERT操作
- [ ] `-D` 删除记录
- [ ] `+U` 更新后的值
- [ ] `-U` 更新前的值



- [ ] Debezium JSON格式 用 `c、u、d、r` 表示操作类型
- [ ] `c`（create）、`u`（update）、`d`（delete）、`r`（read/snapshot）





### 场景题：业务数据库计算的GMV和下游ADS层计算的GMV差5000，如何排查？

这是一个典型的数据对账问题。我的整体思路是 **"分层定位 + 组件归因"** 两步走。

**第一步**，我会先确认统计口径是否一致——很多看板差异其实是口径问题，比如含不含退款、含不含税、订单状态卡在哪个节点。排除口径问题后，我从业务库出发，自顶向下逐层比对：MySQL 真值 → Kafka ODS → Kafka DWD → Doris DWS → ADS 看板。每一层都用 Spark 或 Hive 起离线任务按相同口径聚合，找到**差异第一次出现的层级**。

**第二步**，定位到层级后，再按组件深挖根因。比如：

- 如果差异出现在 ODS → DWD，重点查 Flink 流 Join 维表环节，可能是 HBase 维表延迟、Watermark 把迟到订单丢了、或者 Checkpoint 恢复时数据重放
- 如果差异出现在 DWD → DWS，重点查 Flink → Doris 写入环节，特别是 Aggregate Key 模型的 Stream Load 攒批事务是否完整
- 如果是 ODS 本身就和业务库不一致，那就是 Flink CDC 的位点或 Kafka 的 ack 配置问题

**修复手段**上，如果是实时链路的瞬时问题，靠 CDC 自动捕捉变更回灌；如果是历史数据错误，启动 Spark/Hive 离线任务从源端重算覆盖 Doris 对应分区，也就是我们常说的 **Lambda 架构的离线修补**。



各组件的常见问题清单

| 组件              | 可能的丢数/多数原因                                          | 排查手段                                                     |
| ----------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **Flink CDC**     | 全量阶段切换增量时的位点丢失；DDL 变更未捕获；Snapshot 阶段任务重启 | 查 Flink CDC 的 binlog 位点日志、Checkpoint 状态             |
| **Kafka**         | 生产端 ack 配置不是 `all`、分区副本同步不足；消费端 offset 提交早于处理完成；Topic 保留时间过短数据被清理 | 查 Kafka 的 ISR 列表、`__consumer_offsets`、Producer 端日志  |
| **Flink 计算**    | 维表 Join 失败导致订单被 filter；Watermark 推进过快，迟到数据被丢；Checkpoint 失败回滚导致重算；状态 TTL 过期 | 查 Flink Web UI 的 numRecordsIn/Out、SideOutput 侧输出流、Checkpoint 历史 |
| **HBase（维表）** | 维表数据延迟到达，订单查不到用户信息被过滤；Redis 缓存了脏数据导致 Join 错误 | 抽样订单的 user_id，对比 HBase 实际数据和订单时间戳          |
| **Doris**         | Aggregate Key 模型的 Stream Load 攒批失败，事务回滚但下游以为成功；Unique Key 模型的 replace 把正确数据覆盖成错误的；分区裁剪 SQL 漏掉某个分区 | 查 Doris 的 `show load` 历史、`be.INFO` 日志、`audit log`    |



### Redis 配置策略

- [ ] **开启混合持久化**（`aof-use-rdb-preamble yes`）：兼顾恢复速度和数据安全
- [ ] **AOF 用 everysec 策略**：最多丢 1 秒数据。因为我们是**缓存层**，即使丢了也能通过 Flink CDC 从 MySQL 重建，所以不需要 always
- [ ] **RDB 配置 `save 900 1` + `save 300 10`**：保证至少每 5~15 分钟一次快照，RDB 文件每天定时同步到对象存储做冷备
- [ ] **重写策略**：`auto-aof-rewrite-percentage 100, min-size 256mb`，避免小文件频繁重写
- [ ] 关闭了 **Transparent Huge Pages (THP)**，避免 COW 时 2MB 大页触发大面积复制；早期没关 THP，高峰期 fork 之后主线程出现 500ms+ 的延迟毛刺，影响了 Flink Lookup Join。排查是 COW 在 2MB 大页上放大了复制开销，关掉 THP 后毛刺消失



### Doris 使用

- [ ] 在腾讯实时数仓项目里，Doris 承担 DWS 汇总层和 ADS 应用层的存储和查询。集群部署了 **3 个 FE (1 Leader + 2 Follower) + 10 个 BE**，每个 BE 配 SSD，总存储约 TB 级。
- [ ] **表设计**：DWS 热点词统计用 **Aggregate Key 模型**，按 `event_date` 分区，按 `keyword_id` 分 32 个桶；ADS 看板明细用 **Duplicate Key 模型**，配合**物化视图**预聚合常用维度组合
- [ ] **性能优化**：事实表之间 Join 设计成 **Colocate Group**，避免 Shuffle；维度表 Join 用 **Broadcast Join**
- [ ] **Flink 写入**：用 **Flink Doris Connector** 的 Stream Load 模式，设置 `sink.batch.size=10000` 和 `sink.batch.interval=5s` 攒批写入，避免 Compaction 压力
- [ ] **容量规划**：每个 BE 控制在 **2TB 数据 / 16GB 内存**，Tablet 数量约 5000





### Redis 分布式锁的超卖问题如何解决？

在电商秒杀/抢购等大流量的场景下，使用redis作为缓存，如何保证商品库存不会“超卖”？

- [ ] **原因：** 根本原因是redis异步复制导致的锁信息丢失，Redis 主从复制是异步的，主节点写入成功后还没同步就宕机，从节点提升为主节点后锁信息丢失，第二个客户端可以重新拿到锁，造成超卖
- [ ] **第一层：**库存扣减用 Lua 脚本做原子操作，在 Redis 端一条命令完成"判断库存 + 扣减"
- [ ] **第二层：** 数据库层做最终兜底，用乐观锁（`UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0`），即使 Redis 层出现极端情况，数据库层也能拦住
- [ ] **第三层：**对于主从切换导致的极小概率超卖，生产中通常选择接受，事后通过发优惠券或人工补偿处理；**需要优先保障99%的可用性和吞吐量，用业务手段兜底极端 case**



### 为什么HBase、Doris等列式数据库在查询时不需要回表，而MySQL这样的关系型数据库需要回表？ 

- [ ] MySQL 的回表本质上是 OLTP 场景下**写优化和读优化之间的权衡**
- [ ] MySQL 面向**高频INSERT 和 UPDATE** ，还要保证事务一致性，所以它把**二级索引和数据分离，二级索引只存主键指针**，这样数据更新时只要主键不变，大幅降低了写入的维护开销
- [ ] 代价就是查询时如果二级索引没有覆盖所需的列，就要拿主键回聚簇索引再查一次
- [ ] HBase、Doris 面向**大批量追加写入和聚合分析查询**，不存在频繁更新索引的问题，所以可以**把索引和数据绑在一起**；比如Doris 按 Key 排序后数据和索引天然绑定，HBase 按 RowKey 物理排列直接定位



### Doris 怎么解决 分布式系统中"写成功但确认失败"的问题？

Doris 集群中，BE（后端）已经把数据写成功了，但 FE（前端）因为故障没有收到 BE 的回包，导致 FE 认为这次写入失败。这时候数据到底丢没丢？怎么保证一致性？

- [ ] Doris 内部，FE 通过 **EditLog(WAL)** 做预写日志，**事务的 begin 和 commit 都会先持久化到 EditLog**；BE 写入成功后数据处于 pending 状态，只有 FE 执行 publish 后才对查询可见。
- [ ] 如果 FE 在 commit 之前崩溃，重启后回放 EditLog 发现该事务没有 commit 记录，就会 回滚掉，BE 端的 pending 数据也不会被读到，不会出现脏数据。
- [ ] Flink 侧，我们用 2PC Sink：checkpoint 时 pre-commit 把数据发给 Doris 拿到事务 ID，checkpoint 完成后才正式提交。如果 checkpoint 失败就 回滚
- [ ] 最后 Unique Key 模型做幂等兜底，即使极端情况下重发数据，同 Key 数据后写覆盖，不会重复
