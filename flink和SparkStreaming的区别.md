**如何实现端到端 Exactly-once（精确一次）？**

1. **前提**
   - [ ] **Source：** 支持重新读取数据（如 Kafka 的 Offset）
   - [ ] **Flink 内部：** 依赖 Checkpoint 机制（分布式快照）
   - [ ] **Sink：** 必须支持**事务（Transactions）\**或者\**幂等写入**
2. **第一阶段：预提交**
   - [ ] **Barrier 到达：** 当 JobManager 触发 Checkpoint，Barrier（屏障）流向所有算子
   - [ ] **开启事务：** 当 Sink 算子收到 Barrier 后，它会向外部系统（如 Kafka）开启一个新的事务
   - [ ] **写入数据：** 数据正常写入外部系统，但这些数据在事务中被标记为“未提交” (Uncommitted)。对于 Kafka 消费者来说，如果隔离级别是 **read_committed**，此时是看不到这些数据的
   - [ ] **制作快照：** Sink 算子停止写入，将状态（如事务 ID）**保存到状态后端**，并向 JobManager 报告自己已完成预提交
3. **第二阶段：正式提交**
   - [ ] **通知完成：** 当 JobManager 收到所有算子的成功反馈后，认为这次 Checkpoint 真正成功，会向所有算子发送“确认成功”的指令
   - [ ] **完成事务：** Sink 收到确认指令后，正式提交外部系统的事务（在 Kafka 中标记为 Committed）
   - [ ] **数据可见：** 此时，下游消费者才能读取到这批数据



**容错性：如果在“预提交”阶段失败？**

- [ ] 当前Checkpoint 尚未完成
- [ ] Flink 重启后，会回滚到上一个成功的 Checkpoint
- [ ] 因为之前的事务没提交，外部系统会根据**超时机制**丢弃那部分数据，保证数据不重复



**容错性：如果在“确认成功”指令下达时失败？**

- [ ] 此时 Checkpoint 已经成功，但 Sink 还没来得及提交事务就挂了
- [ ] Flink 重启后，会从**状态中恢复**并发现有一个“**已完成但未提交**”的事务
- [ ] Flink 会根据状态信息，**手动再次触发外部系统的提交操作**，从而保证数据不丢失



**Kafka Sink 的具体实现**

- [ ] **FlinkKafkaProducer**， 由 Flink 提供
- [ ] Kafka 的 transactional.id 必须唯一
- [ ] 消费者的 isolation.level （隔离**级别**）必须设置为 **read_committed**
- [ ] **事务超时时间**（Transaction Timeout）要**大于 Checkpoint 的间隔**，否则事务会在提交前被 Kafka 强制关闭
