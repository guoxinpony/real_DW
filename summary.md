# 1.系统架构

- **数据采集：**使用 **Flume** 实时监控日志文件并将其推送到 **Kafka** 集群。这是非常标准的“日志解耦”做法
- **消息缓冲：** **Kafka** 作为数据总线，解耦了采集端和处理端，起到削峰填谷的作用，划分了 **ODS**（原始数据层）和 **DWD**（明细数据层）
- **实时计算：** **Flink** 是核心引擎，负责数据清洗、转换（ETL）以及将流式数据与维度数据进行关联，采用“**旁路缓存**”模式。Flink 在处理实时流时，优先查询 **Redis**（热数据缓存），如果缓存未命中则查询 **HBase**（全量维度存储）
- **数据存储与应用：** **Doris (StarRocks)**作为最终的 OLAP 引擎，存储查询宽表（DWS/ADS 层）。其强大的聚合和并行查询能力支撑了前端的报表和即席查询





# 2. DIM

##### 1. 选哪个数据库储存维度表

- 选 **Key-Value（KV）类型**的数据库， 而非传统关系型数据库（MySQL/PostgreSQL），因为要根据主键查询，连接事实表和维度表。另外KV 数据库的查询通常是基于 Key 的直接定位，在算法复杂度上趋近于 O(1)
- **Key-Value（KV）类型**的数据库有：Redis、HBase
- 为什么不用Redis：如果维度表非常巨大（例如用户表有数亿甚至**数十亿行**，包含大量画像标签），内存数据库 Redis 成本太高，无法将 所有数据放进内存
- **HBase** 作为基于磁盘（HDFS）的 KV 存储，利用 LSM-Tree 结构，既能存储 **PB 级**的海量数据，又能提供稳定的随机读性能



##### 2. 如何知道数据是维度表数据

- 在程序中用一个列表储存所有维度表的表名，可以实现，但不灵活，后期不便于更改
- 在配置文件中用一个列表储存所有维度表的表名，不适合分布式系统
- Redis：用一张表储存维度表信息；存在flink连接redis频繁以及redis最终一致性问题
- **Flink CDC:** 立刻捕获维度表状态的变化，将维度表广播到所有flink并行度

##### 3. 实时数仓维度数据同步架构实现原理

- **Maxwell** 实时监听 MySQL 的 **binlog**，将业务数据和维度表名称同步到Kafka不同的主题
- **Flink** 从配置 Topic 读取信息并进行“广播”， 当 MySQL 中的维表配置发生变化时，Flink 作业无需重启即可动态感知； 从 topic_db **读取业务数据**，根据广播出来的配置信息，判断当前数据是否属于目标维度表
- 经过过滤和转换后的维度数据，实时写入 **HBase**。HBase 因其**高并发查询**和**海量存储**能力，常在实时数仓中充当**维度表（DIM 层）**的存储介质，方便后续在 **DWD 层流表关联**（Lookup Join）时使用
- **缺点：** 数据采集端需要维护maxwell这个额外组件，所以一个新方案是使用 **Flink CDC** 捕获变化数据



![FlinkCDC](pic/FlinkCDC.png)

##### 4. 数据同步架构从**Maxwell + Kafka** 切换为 **Flink CDC**

- **简化架构：**去除Maxwell 和对应的 Kafka Topic，使用 Flink CDC 直接与 MySQL 通信，减少了链路中的故障点和Kafka 的存储开销
- **全量与增量的一体化同步：** 之前可能需要手动触发一次业务库查询，现在 Flink CDC 启动时会**自动扫描配置表存量**数据，完成后自动切入 Binlog 监听，确保配置信息始终完整
- **端到端一致性：** Flink CDC 的 Offset 信息直接存储在 Flink 的 Checkpoint 中



##### 5. DIM层流程

**第一阶段：**

1. 环境准备
2. **Checkpoint** 配置：开启 Checkpoint，设置超时时间、最小间隔及取消作业后是否保留检查点

**第二阶段：主流业务数据接入**

1. 从 Kafka 读取主流数据，将读取的数据封装为 **DataStream** 流
2. 数据格式预处理，JSON 字符串 -> JSON 对象，方便访问属性

**第三阶段：配置流接入与动态管理**

1. 使用 **Flink CDC** 读取配置表，实时监控维度配置表的变动，将配置数据封装为 DataStream 流
2. 配置流数据转换，将 CDC 读取的 JSON 字符串转换为实体类对象
3. **HBase 物理表维护**，根据配置表中信息，在 HBase 中执行**建表**或**删表**操作
4. 配置流广播化，将配置流定义为**广播流（Broadcast Stream）**，以便所有并行算子都能实时获取最新的维度配置

**第四阶段：核心逻辑处理（双流关联）**

1. **流关联（Connect）**,  将主流业务数据与广播后的配置流进行 connect 操作
2. 根据广播配置表过滤，将接收到的配置数据存入**广播状态（BroadcastState）**，Key 为维度表名，Value 为配置详情；判断当前数据所属表是否为维度表。若是维度表，则向下游分发；若否，则过滤

**第五阶段：维度表持久化**

1. 写入 HBase， 将经过过滤后的维度数据实时同步到 HBase 对应的物理表中，作为 **DIM 层**存储



# 2. DWD

##### 1. 不同日志数据的分流：页面日志，错误日志，启动日志，曝光日志，动作日志

1. **数据清洗与预处理 (ETL)**

   - [ ] **格式转换：**JSON String -> JSONObject，方便按键值对操作
   - [ ] **脏数据隔离**：利用 **侧输出流 (Side Output)** 将解析失败（非标准 JSON）的数据单独写入特定的 **Kafka 脏数据主题**，防止由于一条垃圾数据导致整个 Flink 作业崩溃

2. **状态修复**

   - [ ] **逻辑**：由于前端埋点不一定准确（如卸载重装），需要利用 Flink 的 **ValueState**

   - [ ] **举例** 如何判断一个用户是**新用户还是老用户**？确保同一设备的数据进入同一个算子实例，查询该设备在**状态后端**是否有访问记录，若状态为空且日志标记为新访客，则更新状态并保持标记；若状态不为空且日志标记为新访客，则将标记强制修正为“老访客”并更新状态；若日志标记为老访客，则只更新状态

   - [ ] 

   - | **场景**       | **日志标记 (is_new)** | **状态后端 (State)** | **Flink 处理动作**    | **最终输出标记** |
     | -------------- | --------------------- | -------------------- | --------------------- | ---------------- |
     | **真新用户**   | `1` (新)              | **空 (Null)**        | 写入当前日期到状态    | `1` (保持)       |
     | **卸载重装**   | `1` (新)              | **非空**             | 无需更新或仅刷新时间  | `0` (**修正**)   |
     | **常规老用户** | `0` (老)              | **非空**             | 刷新状态中的访问时间  | `0` (保持)       |
     | **历史老用户** | `0` (老)              | **空 (Null)**        | **补全状态** (关键点) | `0` (保持)       |

3. **核心分流机制**：**ProcessFunction + Side Output** 模式

   - [ ] **主流 (Main Stream)**：通常选择流量最大的 **页面日志 (Page Log)** 作为主流
   - [ ] **侧输出流 (Side Streams)**：根据业务字段判断，将其他日志数据打上不同的标签（OutputTag）输出，如 **错误日志**：包含 `err` 字段
   - [ ] **优势**：相比多次 filter（过滤器），这种方式只需要对数据进行**一次遍历**，极大提升了处理性能

4. **数据多下游写入 (Kafka Sinks)**

   - [ ] 将拆分后的多条流分别对接到不同的 Kafka Topic 中，完成 DWD 层的构建



##### 2. 业务数据表：评论事实表

1. **工作流程：**

   - 从kafka 业务数据主题读取业务数据

   - 过滤出评论数据，创建评论事实数据流（表）

   - 从Hbase 读取维度表（例如好 中 差评）

   - 关联评论数据和维度表: **lookup join**

   - 输出到 kafka 主题 评论事实表

     

2. **为什么要用lookup join而不是其他类型的JOIN 来关联评论数据表和维度表？**

   - [ ] **Regular Join：** Flink 会将参与 Join 的两侧数据（评论流和维度流）**全部保存在状态（State）中**。当任意一边有新数据进入时，都会去对侧的状态中查找匹配项。维度表（如用户表、商品字典）通常非常庞大，会导致 **Checkpoint 极慢、内存溢出（OOM）**

   - [ ] **Window Join (窗口 Join)：** 两条流的数据必须落在**同一个时间窗口**内，且 Key 相同才能关联成功；维度数据（如字典）通常是**持久存在**的，而评论数据是**瞬时发生**的，导致**生命周期错位**

   - [ ] **Interval Join (间隔 Join)：** 设定一个相对的时间范围。例如：要求评论的时间戳 $t_1$ 必须在维度变更时间 $t_2$ 的 $[t_2 - 1h, t_2 + 1h]$ 范围内； Interval Join 适用于“**订单关联支付**”这种两个事件**都有明确发生时间的场景**。而**维表没有一个严格的业务时间差限制** 

   - [ ]  **Lookup Join (维表关联)：** 当评论流（事实表）进来一条数据时，Flink **主动去外部存储**（如 HBase、MySQL、Redis）查询一次对应的维度数据。**Flink 只查不存，按需点查，高性能优化（异步，LRU, TTL）**

     

3. **用Lookup Join的优点以及优化**

   - [ ] **轻量化：** Flink 状态里**不存维表**，只查不存，完美避开“状态爆炸”

   - [ ] **海量支持：** 维度表可以是 10 亿级的 HBase 表，Flink 只需要按需点查

   - [ ] **异步支持：** 并发发送多个查询请求，极大提升查询吞吐量

   - [ ] **LRU 缓存：** 将频繁访问的维度数据缓存在内存中，减少对外部数据库的压力

   - [ ] **TTL：** 设置缓存过期时间，兼顾性能与数据准确性

     

4. **flink频繁查询HBase，有没有频繁创建和销毁连接？**

   - [ ] **没有，**Flink 都采用了**单次初始化**的设计

   - [ ] 通过**RichFunction**接口管理生命周期，在**open()** 方法**初始化长连接**以连接外部数据库，在**eval() / map()** 方法**处理数据**，作业停止或发生故障重启时，才会调用 **close()** 来正常销毁连接

   - [ ] **HBase 客户端连接池机制：** HBase 客户端共享 Connection 对象，Connection 对象是重量级的、线程安全的；HBase 客户端在**第一次查询**时会去 ZooKeeper 寻找 Meta 表并**缓存** Region 的位置

   - [ ] **Flink Connector 异步 IO：** 在一个长连接（或连接池）上同时发出多个请求，而不需要等待前一个请求返回

     

5. **富函数richfunction：算子的生命周期管理**

   - [ ] **初始化 ：** open()在算子的每个并行实例启动时，**仅调用一次**；用于进行**开销较大**的初始化操作，比如 **建立长连接**，**初始化 LRU 缓存**
   - [ ] **数据处理：** map()等算子每流入一条数据，就会调用一次；**直接**复用 open() 中建立好的连接对象
   - [ ] **清理：** close() 在作业正常停止、取消或发生异常关闭时，**仅调用一次**；**关闭数据库连接**，**清空并关闭缓存**，**刷写（Flush）**
