# 1.系统架构

- **数据采集：**使用 **Flume** 实时监控日志文件并将其推送到 **Kafka** 集群。这是非常标准的“日志解耦”做法
- **消息缓冲：** **Kafka** 作为数据总线，解耦了采集端和处理端，起到削峰填谷的作用，划分了 **ODS**（原始数据层）和 **DWD**（明细数据层）
- **实时计算：** **Flink** 是核心引擎，负责数据清洗、转换（ETL）以及将流式数据与维度数据进行关联，采用“**旁路缓存**”模式。Flink 在处理实时流时，优先查询 **Redis**（热数据缓存），如果缓存未命中则查询 **HBase**（全量维度存储）
- **数据存储与应用：** **Doris (StarRocks)**作为最终的 OLAP 引擎，存储查询宽表（DWS/ADS 层）。其强大的聚合和并行查询能力支撑了前端的报表和即席查询





# 2. DIM

##### 1. 选哪个数据库储存维度表

- 选 **Key-Value（KV）类型**的数据库， 而非传统关系型数据库（MySQL/PostgreSQL），因为要根据主键查询，连接事实表和维度表。另外KV 数据库的查询通常是基于 Key 的直接定位，在算法复杂度上趋近于 O(1)
- **Key-Value（KV）类型**的数据库有：Redis、HBase
- 为什么不用Redis：如果维度表非常巨大（例如用户表有数亿甚至**数十亿行**，包含大量画像标签），内存数据库 Redis 成本太高，无法将 所有数据放进内存
- **HBase** 作为基于磁盘（HDFS）的 KV 存储，利用 LSM-Tree 结构，既能存储 **PB 级**的海量数据，又能提供稳定的随机读性能



##### 2. 如何知道数据是维度表数据

- 在程序中用一个列表储存所有维度表的表名，可以实现，但不灵活，后期不便于更改
- 在配置文件中用一个列表储存所有维度表的表名，不适合分布式系统
- Redis：用一张表储存维度表信息；存在flink连接redis频繁以及redis最终一致性问题
- **Flink CDC:** 立刻捕获维度表状态的变化，将维度表广播到所有flink并行度

![2_2 dim_table](pic/2_2dim_table.png)



##### 3. 实时数仓维度数据同步架构实现原理

- **Maxwell** 实时监听 MySQL 的 **binlog**，将业务数据和维度表名称同步到Kafka不同的主题
- **Flink** 从配置 Topic 读取信息并进行“广播”， 当 MySQL 中的维表配置发生变化时，Flink 作业无需重启即可动态感知； 从 topic_db **读取业务数据**，根据广播出来的配置信息，判断当前数据是否属于目标维度表
- 经过过滤和转换后的维度数据，实时写入 **HBase**。HBase 因其**高并发查询**和**海量存储**能力，常在实时数仓中充当**维度表（DIM 层）**的存储介质，方便后续在 **DWD 层流表关联**（Lookup Join）时使用
- **缺点：** 数据采集端需要维护maxwell这个额外组件，所以一个新方案是使用 **Flink CDC** 捕获变化数据



![FlinkCDC](pic/FlinkCDC.png)

##### 4. 数据同步架构从**Maxwell + Kafka** 切换为 **Flink CDC**

- **简化架构：**去除Maxwell 和对应的 Kafka Topic，使用 Flink CDC 直接与 MySQL 通信，减少了链路中的故障点和Kafka 的存储开销
- **全量与增量的一体化同步：** 之前可能需要手动触发一次业务库查询，现在 Flink CDC 启动时会**自动扫描配置表存量**数据，完成后自动切入 Binlog 监听，确保配置信息始终完整
- **端到端一致性：** Flink CDC 的 Offset 信息直接存储在 Flink 的 Checkpoint 中



##### 5. DIM层流程

**第一阶段：**

1. 环境准备
2. **Checkpoint** 配置：开启 Checkpoint，设置超时时间、最小间隔及取消作业后是否保留检查点

**第二阶段：主流业务数据接入**

1. 从 Kafka 读取主流数据，将读取的数据封装为 **DataStream** 流
2. 数据格式预处理，JSON 字符串 -> JSON 对象，方便访问属性

**第三阶段：配置流接入与动态管理**

1. 使用 **Flink CDC** 读取配置表，实时监控维度配置表的变动，将配置数据封装为 DataStream 流
2. 配置流数据转换，将 CDC 读取的 JSON 字符串转换为实体类对象
3. **HBase 物理表维护**，根据配置表中信息，在 HBase 中执行**建表**或**删表**操作
4. 配置流广播化，将配置流定义为**广播流（Broadcast Stream）**，以便所有并行算子都能实时获取最新的维度配置

**第四阶段：核心逻辑处理（双流关联）**

1. **流关联（Connect）**,  将主流业务数据与广播后的配置流进行 connect 操作
2. 根据广播配置表过滤，将接收到的配置数据存入**广播状态（BroadcastState）**，Key 为维度表名，Value 为配置详情；判断当前数据所属表是否为维度表。若是维度表，则向下游分发；若否，则过滤

**第五阶段：维度表持久化**

1. 写入 HBase， 将经过过滤后的维度数据实时同步到 HBase 对应的物理表中，作为 **DIM 层**存储



# 3. DWD

### 1. 不同日志数据的分流：页面日志，错误日志，启动日志，曝光日志，动作日志

1. **数据清洗与预处理 (ETL)**

   - [ ] **格式转换：**JSON String -> JSONObject，方便按键值对操作
   - [ ] **脏数据隔离**：利用 **侧输出流 (Side Output)** 将解析失败（非标准 JSON）的数据单独写入特定的 **Kafka 脏数据主题**，防止由于一条垃圾数据导致整个 Flink 作业崩溃

2. **状态修复**

   - [ ] **逻辑**：由于前端埋点不一定准确（如卸载重装），需要利用 Flink 的 **ValueState**

   - [ ] **举例** 如何判断一个用户是**新用户还是老用户**？确保同一设备的数据进入同一个算子实例，查询该设备在**状态后端**是否有访问记录，若状态为空且日志标记为新访客，则更新状态并保持标记；若状态不为空且日志标记为新访客，则将标记强制修正为“老访客”并更新状态；若日志标记为老访客，则只更新状态

   - [ ] 

   - | **场景**       | **日志标记 (is_new)** | **状态后端 (State)** | **Flink 处理动作**    | **最终输出标记** |
     | -------------- | --------------------- | -------------------- | --------------------- | ---------------- |
     | **真新用户**   | `1` (新)              | **空 (Null)**        | 写入当前日期到状态    | `1` (保持)       |
     | **卸载重装**   | `1` (新)              | **非空**             | 无需更新或仅刷新时间  | `0` (**修正**)   |
     | **常规老用户** | `0` (老)              | **非空**             | 刷新状态中的访问时间  | `0` (保持)       |
     | **历史老用户** | `0` (老)              | **空 (Null)**        | **补全状态** (关键点) | `0` (保持)       |

3. **核心分流机制**：**ProcessFunction + Side Output** 模式

   - [ ] **主流 (Main Stream)**：通常选择流量最大的 **页面日志 (Page Log)** 作为主流
   - [ ] **侧输出流 (Side Streams)**：根据业务字段判断，将其他日志数据打上不同的标签（OutputTag）输出，如 **错误日志**：包含 `err` 字段
   - [ ] **优势**：相比多次 filter（过滤器），这种方式只需要对数据进行**一次遍历**，极大提升了处理性能

4. **数据多下游写入 (Kafka Sinks)**

   - [ ] 将拆分后的多条流分别对接到不同的 Kafka Topic 中，完成 DWD 层的构建



### 2. 业务数据表：评论事实表

1. **工作流程：**

   - [ ] 从Kafka 业务数据主题读取业务数据

   - [ ] 过滤出评论数据，创建评论事实数据流（表）

   - [ ] 从HBase 读取维度表（例如好 中 差评）

   - [ ] 关联评论数据和维度表: **look-up join**

   - [ ] 输出到 Kafka 主题 评论事实表

     

2. **为什么要用lookup join而不是其他类型的JOIN 来关联评论数据表和维度表？**

   - [ ] **Regular Join：** Flink 会将参与 Join 的两侧数据（评论流和维度流）**全部保存在状态（State）中**。当任意一边有新数据进入时，都会去对侧的状态中查找匹配项。维度表（如用户表、商品字典）通常非常庞大，会导致 **Checkpoint 极慢、内存溢出（OOM）**

   - [ ] **Window Join (窗口 Join)：** 两条流的数据必须落在**同一个时间窗口**内，且 Key 相同才能关联成功；维度数据（如字典）通常是**持久存在**的，而评论数据是**瞬时发生**的，导致**生命周期错位**

   - [ ] **Interval Join (间隔 Join)：** 设定一个相对的时间范围。例如：要求评论的时间戳 $t_1$ 必须在维度变更时间 $t_2$ 的 $[t_2 - 1h, t_2 + 1h]$ 范围内； Interval Join 适用于“**订单关联支付**”这种两个事件**都有明确发生时间的场景**。而**维表没有一个严格的业务时间差限制** 

   - [ ]  **Lookup Join (维表关联)：** 当评论流（事实表）进来一条数据时，Flink **主动去外部存储**（如 HBase、MySQL、Redis）查询一次对应的维度数据。**Flink 只查不存，按需点查，高性能优化（异步，LRU, TTL）**

     

3. **用Lookup Join的优点以及优化**

   - [ ] **轻量化：** Flink 状态里**不存维表**，只查不存，完美避开“状态爆炸”

   - [ ] **海量支持：** 维度表可以是 10 亿级的 HBase 表，Flink 只需要按需点查

   - [ ] **异步支持：** 并发发送多个查询请求，极大提升查询吞吐量

   - [ ] **LRU 缓存：** 将频繁访问的维度数据缓存在内存中，减少对外部数据库的压力

   - [ ] **TTL：** 设置缓存过期时间，兼顾性能与数据准确性

     

4. **flink频繁查询HBase，有没有频繁创建和销毁连接？**

   - [ ] **没有，**Flink 都采用了**单次初始化**的设计

   - [ ] 通过**RichFunction**接口管理生命周期，在**open()** 方法**初始化长连接**以连接外部数据库，在**eval() / map()** 方法**处理数据**，作业停止或发生故障重启时，才会调用 **close()** 来正常销毁连接

   - [ ] **HBase 客户端连接池机制：** HBase 客户端共享 Connection 对象，Connection 对象是重量级的、线程安全的；HBase 客户端在**第一次查询**时会去 Zookeeper 寻找 Meta 表并**缓存** Region 的位置

   - [ ] **Flink Connector 异步 IO：** 在一个长连接（或连接池）上同时发出多个请求，而不需要等待前一个请求返回

     

5. **富函数richfunction：算子的生命周期管理**

   - [ ] **初始化 ：** open()在**算子的每个并行实例**启动时，**仅调用一次**；用于进行**开销较大**的初始化操作，比如 **建立长连接**，**初始化 LRU 缓存**

   - [ ] **数据处理：** map()等算子每流入一条数据，就会调用一次；**直接**复用 open() 中建立好的连接对象

   - [ ] **清理：** close() 在作业正常停止、取消或发生异常关闭时，**仅调用一次**；**关闭数据库连接**，**清空并关闭缓存**，**刷写（Flush）**

     

6. **如何保证Flink连接HBase可靠性**

   - [ ] **心跳检测：** 在 `map()` 里查询前，先简单判断下 `conn.isValid()`
   - [ ] **自动重连：** 如果发现断了，在 `map()` 内部触发一个小规模的重新初始化逻辑
   - [ ] **连接池：** 很多时候我们不在 `open()` 里创建单个连接，而是创建一个**连接池（Connection Pool）**。池子会自动管理连接的存活和失效重连

# 4. DWS

### 1. 作用

- 汇总表抽取和数据轻度聚合
- **汇总表抽取：**例如在ADS层需要统计：当日各渠道独立访客数，当日各渠道平均会话时长，当日各渠道会话总数，当日各渠道每会话平均浏览页面数量
- 则在DWS层抽取为一张表，包含：时间，渠道，独立访客，会话数量，页面浏览量，访问时长
- **数据轻度聚合：** 列式储存数据库



### 2. 选哪个数据库储存DWS表

1. **为什么选择列式储存数据库，而不是行式储存** 
   - [ ] **极高的读取效率（降低磁盘I/O）：** 电商 DWS 层包含是“**宽表**”，**有数百列指标**，**列存**只读特定列数据，跳过其他列；**行存**通过主键读取整行所有字段，造成了巨大的无效 I/O
   - [ ] **极致的数据压缩比：** 同一列的数据类型相同且相似度极高，相比于行式储存的一行数据包括不同字段类型，压缩率更高
   - [ ] **向量化执行，计算性能更高：** CPU 可以一次性**处理一批列**数据，而不是一条条地处理行数据。“向量化执行”将聚合计算（SUM/COUNT）的**性能提升了 10 倍**以上
2. **列式数据库有HBase, Doris, clickhouse, 为什么选doris**
   - [ ] HBase 物理上虽然按列族存储，但在同一个列族内，它是按照 **RowKey 顺序存储的一行**行数据，本质上更接近于行存的变种，面向**随机读写优化**
   - [ ] Hbase **缺乏列式计算引擎**（如向量化执行、谓词下推)，在做大规模 SUM 或 GROUP BY 时，性能远不如专门的实时 OLAP 引擎 Doris
   - [ ] 因此，HBase 在实时数仓中更多承担 **DWD 明细层存储**或 **ADS 层的点查服务**
   - [ ] **ClickHouse**对数据的**异步更新和删除**，存在**延迟**且极其消耗资源，**很难处理“实时累加指标”**的频繁变动，而**Doris** 拥有强大的 **Primary Key 模型**，支持**高性能的实时更新**
   - [ ] **ClickHouse多表JOIN能力弱**，分布式 Join 的实现复杂且容易 OOM；Doris支持 Shuffle Join能轻松应对电商中复杂的维度关联查询
   - [ ] “**HBase 强在点查，ClickHouse 强在单表极速扫描，而 Doris 强在‘实时更新+复杂关联+极低运维代价’**



### 3. DWS表：搜索词频率统计

1. **工作流程**

   - [ ] 从页面日志读取数据 (DWD层)

   - [ ] 过滤搜索行为，排除非搜索点击

   - [ ] 分词

   - [ ] 开窗与聚合：轻度聚合，统计当前时间范围内搜素词热度

   - [ ] 写入 Doris：**Aggregate Key** 聚合模型，关键词、时间窗口作为 Key，出现次数作为 Value，设置聚合类型为 SUM

     

2. **数据迟到Allowed Lateness和乱序处理Watermark**

   - [ ] **Watermark = 5 s,** 相当于容忍时间为 5秒 的乱序策略，**Watermark**决定窗口**何时被触发进行第一次计算**

   - [ ] **处理空闲数据源：** 由于有多个并行度， Flink 的 Watermark 是以所有**上游分区中最小**的那个为准，如果某个分区长时间不更新，会导致整个**窗口无法触发，数据卡死**

   - [ ] **Allowed Lateness = 1 min**：决定窗口**状态（State）保留多久**, 在 Watermark 触发窗口后，如果又有迟到的数据进入这个窗口，Flink 会利用之前的**状态**重新计算，并输出一个更新之后的结果

   - [ ] Watermark 设为 5 秒（保证快速响应，低延迟），Allowed Lateness 设为 1 分钟（保证最终准确性）；**第一次**Watermark 超过窗口结束时间，输出初步统计结果；**后续**每来一条迟到数据，Flink 窗口都会**重新计算**一次，并再次向输出（Sink）发出一条结果

   - [ ] **侧输出流**：对于超过 5 分钟还没到的极其严重的迟到数据，写入离线表另行计算

     



### 4.  DWS表：独立访客数uv

**工作流程**

- [ ] 从动作日志读取数据，过滤掉没有 userId 的非法请求

- [ ] 指定watermark

- [ ] Flink内去重，在 Flink 算子内配合**布隆过滤器**进行实时去重

- [ ] 开窗，聚合，将结果写到Doris表中

  

**去重实现1：状态 \+ 定时器**

- [ ] **KeyBy 分组**：按统计维度（如：页面 ID 或“全站”）分组

- [ ] **状态存储 (State)**：使用 MapState<String, Boolean> 或 ValueState< BloomFilter> 来记录已经访问过的用户 ID

- [ ] **去重判断**：每来一条数据，检查状态中是否存在该userId，如果不存在：说明是新访客，计数 +1，并将 ID 存入状态

- [ ] **定时器清空 (Timer)**：为了防止状态无限增长，需要注册一个**天级定时器**（例如在每天 00:00:00），触发时清空状态，开启新一天的统计

- [ ] **布隆过滤器优化：** 直接存 userId 到 MapState 会OOM，极大地节省空间

  

**去重实现2：状态 \+ 撤回**

- [ ] 常出现在 Flink SQL 的 **COUNT(DISTINCT userId)**

- [ ] **记录状态 (State Management)**：算子必须记录已经处理过的明细数据统计值，目的是判断新到的数据是“新访客”还是“老用户更新”

- [ ] **触发判断 (Trigger Logic)**：当一条**新数据进入**时，查询状态，如果发现**状态需要改变**（例如 UV 从 100 变 101），则进入**撤回**流程

- [ ] **发送撤回消息 (Retract Old Value)**：向下游发送一条类型为 `UPDATE_BEFORE` 或 `DELETE` 的消息，数值通常为旧结果的**负值**

- [ ] **发送更新消息 (Accumulate New Value)**：向下游发送一条类型为 `UPDATE_AFTER` 或 `INSERT` 的消息，数值为**最新计算的结果**

  

**如果下游系统（如 Kafka）不支持撤回消息怎么办？**

- [ ] 使用 **Changelog 格式**下游完整保留“撤回”这个动作，在定义 Sink 时，指定格式为 `debezium-json`，Flink 会将每一行包装成一个**特殊的 JSON**，里面包含一个 `op` 字段，例如**"op": "d" (Delete) 和"op": "u" (Update)**，这样下游能通过 **op 字段**感知到这是一条撤回消息，从而在**数据库层面**执行 DELETE 或 REPLACE

- [ ] **Upsert Kafka Connector：将流转换为“按 Key 更新”的格式**，当 Flink 发送 **+I 或 +U** 时，它向 Kafka **发送普通的 Value**；当 Flink 发送 **-D 或 -U** 时，它向 Kafka 发送一条 **Value 为 Null** 的消息（称为 **Tombstone 墓碑消息**）

  

**如何选择 “状态 + 定时器 + 布隆过滤器” 和 “状态 + 撤回” 两种思路？**

- [ ] **状态 + 撤回** （Flink SQL）**COUNT(DISTINCT)** 适合日活在**百万级以下**，在这个量级下，内存压力尚在可控范围内，SQL 的开发效率优势远大于内存优化

- [ ] **状态 + 定时器 + 布隆过滤器**适合日活达到**千万级甚至亿级**，如果用户量巨大，**状态**（State）会极速膨胀，导致 Checkpoint 变慢甚至 OOM；配合 **布隆过滤器**，将数 GB 的状态压缩至几百 MB

- [ ] **状态 + 撤回** 使用Flink SQL **开发效率极高**，一行 SQL 搞定去重和撤回逻辑

- [ ] **状态 + 定时器** **极高的控制力**，可以精确控制什么时候清空状态、什么时候输出结果

- [ ] 输出下游**支持撤回**如ClickHouse，Redis，**使用状态 + 撤回**，利用这些数据库的去重引擎来对接 Flink 的撤回流

- [ ] 输出下游**不支持撤回**如Kafka，**使用状态 + 定时器**，可以手动控制在定时器触发时只下发一个最终的统计值，规避掉“一正一负”的撤回波

  

**布隆过滤器的容量预估** 
$$
m = -\frac{n*lnp}{{(ln2)}^2}
$$

- n：预期的去重数据量（如 1000 万）
- p：容忍的误判率（如 0.01）
- m：需要的比特位数量
- 对于日活 1000 万的用户，误判率设为 1% 时，布隆过滤器大约占用 95MB 左右的内存。这比直接存 1000 万个 String（约几百 MB 到 GB 级别）要节省得多
- **追问 A：布隆过滤器存在状态里，Checkpoint 压力大怎么办？** 如果 ValueState 存布隆过滤器导致 CP 几十 GB，说明不适合存 Flink 状态; 改用 **Redis 的 BitMap**。Flink 只负责计算 Hash 值并发送给 Redis，利用 Redis 的 **SETBIT** 指令。这样 Flink 状态极轻，压力转移到了专门做内存存储的 Redis 上
- **追问 B：定时器触发的一瞬间，如果正好有数据进入怎么办？** 在 Key 后面加上日期（如 `2026-01-13_page01`）。这样即使跨天，旧数据也会落在旧 Key 的定时器里处理，新数据落在新 Key 里，互不干扰
- **追问 C：布隆过滤器有误判，如何实现“完全精准”且“超大规模”去重？** **RoaringBitmap** 比普通 BitMap 更省空间，且它是**精确去重**。目前主流的 OLAP 引擎（如 **Doris**, StarRocks）底层都用它



### 5. DWS表：去重，如果处理超大规模实时 UV 统计，状态 + 定时器 + 布隆过滤器不足以应对，该怎么办？

1. **思路：**

   - [ ] **Redis BitMap 外部去重**
   - [ ] **Flink **接收明细数据、生成 Hash 值、异步请求 Redis、下发统计结果，**Redis **利用位图（BitMap）存储用户访问标记

2. **实现：**

   - [ ] **偏移量转换：**Redis 的 BitMap 是基于 String 类型的位操作，它需要一个 **Integer 类型的偏移量**，如果数字类型，直接作为 Offset；如果字符串，需要通过 Hash 算法映射为数字
   - [ ] **利用 SETBIT 的原子性：** SETBIT 命令会返回该位在设置之前的旧值，如果返回 0，说明该位之前是 0，这是**新访客**；如果返回 1，说明该位之前就是 1，这是**重复访客**
   - [ ] **定时清空与 Key 设计：** **uv:project_name:yyyy-MM-dd:page_id**，通过 Redis 的 EXPIRE 设置为 24~48 小时，自动清理历史数据，无需 Flink 定时器维护
   - [ ] **优化：异步 I/O**，如果每来一条数据都同步请求 Redis，Flink 的吞吐量会由于**网络 IO 延迟**而急剧下降；使用异步IO，Flink 可以在**等待 Redis 返回结果**的同时**处理下一条数据**，**并发执行 IO** 请求，极大提升了吞吐性能

3. **面试总结**

   在处理**超大规模全站 UV** 时，我会考虑将去重逻辑外置到 **Redis BitMap**。

   理由是这种方式可以将 Flink 的状态压力转移到 Redis，保证了 Checkpoint 的秒级完成，且方便其他业务系统直接查询。

   在实现上，我会对 `userId` 进行 Hash 映射得到 Offset，利用 Redis `SETBIT` 的原子返回值判断是否为新用户。同时，为了解决网络延迟问题，我会采用 **Flink Async I/O** 机制进行异步读写，确保单核支撑万级以上的 TPS。



# 5. DIM表：优化查询维度表 旁路缓存

### 1. 查询逻辑

- [ ] Flink Job 先根据 Key 查询 Redis
- [ ]  如果 Redis **命中（Hit）**，直接返回数据，处理过程结束
- [ ] 如果 Redis **缺失（Miss）**，则发起对 HBase 的请求
- [ ]  从 HBase 获取数据后，将其回写到 Redis，并设置一个 **过期时间（TTL）**，以便下次查询直接命中



### 2. 为什么需要旁路缓存

| **维度**       | **HBase 直接查询**            | **Redis + HBase 旁路缓存**        |
| -------------- | ----------------------------- | --------------------------------- |
| **查询延迟**   | 毫秒级（通常 5-20ms）         | 微秒级（通常 <1ms）               |
| **并发支撑**   | 受限于 RegionServer 负载      | 极高，单机支持万级甚至十万级 QPS  |
| **HBase 压力** | 高并发下 HBase 压力大，易抖动 | 显著降低 HBase IOPS，保护底层存储 |



### 3. 如何解决缓存一致性问题，hbase和redis中数据可能不一致？

1. 设置合理**TTL**
   - [ ] 实现简单
   - [ ] 可能存在**不一致窗口期**，在 TTL 到期前，Redis 里的数据始终是旧的，保证**最终一致性**
   - [ ] 设置 TTL 时增加一个**随机秒数**，防止大量缓存在同一时刻失效，造成**缓存雪崩**
2. **双写**：先更新 HBase，再删除 Redis
   - [ ] 下次查询时，Redis 缺失，重新从 HBase 加载最新数据
   - [ ] 为什么删除redis旧数据而不是更新：更新 Redis 涉及**并发**写竞争，容易导致“**脏数据**”覆盖
   - [ ] 由于**分布式事务的原子性（Atomicity）缺失**， 可能存在**HBase 成功但 Redis 失败**的情况
   - [ ] **延迟双删**： 在**更新 HBase 前删一次** Redis，更新完后，**睡眠**一段时间（如 500ms）再删一次，解决**并发脏数据回写**
   - [ ] 睡眠会导致**吞吐量骤降**
3. **Flink CDC 实时同步**
   - [ ] 使用 Flink CDC 监听 HBase WAL 变更，一旦有变动，Flink 任务实时更新或删除 Redis 中的对应 Key
   - [ ] 即使维度更新任务异常，同步任务也可以通过 Checkpoint 保证最终一致
   - [ ] 需要额外部署 Flink CDC 同步链路
   - [ ] **解决redis部分失败：** 基于数据库日志（WAL/Binlog）。只要 HBase 写入成功并记录了日志，CDC 就一定会捕捉到这个变更
   - [ ] **解决阻塞与休眠：** 延迟双删为了解决并发脏数据回写，通常需要 `Thread.sleep(500ms)`，阻塞 TaskManager 的处理线程；**Flink CDC**是**异步、非阻塞**的，不干扰Hbase写入的性能



# 6. DWD表： 优化吞吐量 异步I/O

### 1. 为什么需要异步 I/O

1.  解决**吞吐量小**的问题
   - [ ] 同步查询会导致严重的 **I/O 等待**
   - [ ] 假设一次数据库查询耗时 10ms，那么单个线程每秒只能处理 100 条数据
   - [ ] 导致数据堆积和反压
2. 提高**资源利用率**
   - [ ] 异步 I/O 允许一个线程在**等待数据库响应**的同时，继续**发送数据的查询请求**
   - [ ] 单机吞吐量提升数倍甚至数十倍

| **特性**       | **同步 I/O (Sync)**        | **异步 I/O (Async)**             |
| -------------- | -------------------------- | -------------------------------- |
| **等待方式**   | 阻塞当前线程，直到结果返回 | 非阻塞，发送请求后立即处理下一个 |
| **吞吐量**     | 低，受限于单次请求延迟     | 高，受限于并发数和网络带宽       |
| **数据库压力** | 压力较小且平稳             | 瞬间并发压力可能较大（需限流）   |
| **适用场景**   | 低频数据、极简单的逻辑     | 高频实时流、海量维表关联         |



### 2. Flink 如何实现异步 I/O 

1. **实现 `AsyncFunction`**：定义如何发送异步请求及如何处理回调
2. **使用异步客户端**：必须使用支持异步查询的数据库驱动（如 Lettuce 的 Redis 客户端、HBase 的异步 API）
3. **应用异步转换**：在 DataStream 上调用 `AsyncDataStream.unorderedWait` 或 `orderedWait`
4. **unorderedWait：** 只要请求返回，就**立刻向下游发送**数据。**性能最高**，但会导致数据**乱序**
5. **orderedWait**：保证输出顺序与输入**顺序一致**。通过缓存后续到达的结果来实现，**性能略低**，且可能增加**延迟**
