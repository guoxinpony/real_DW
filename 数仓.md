### 数据仓库如何分层，为什么要分层？

##### 如何分层：

- [ ] **ODS：**结构与业务数据库基本一致。主要作用是**数据备份**、抗压（**不直接查业务库**）和**溯源**
- [ ] **DWD：** 清洗与规范化： 进行**清洗**（判空/去重）、脱敏、**维度退化**（宽表化）。这一层通常是**最细粒度**的事实表
- [ ] **DIM:**  提供**业务上下文**，构建**一致性维度**
- [ ] **DWS：** 预计算： 按主题（如：天、用户、商品）进行**轻度聚合**。存放通用的统计指标，提升查询效率，减少重复计算
- [ ] **ADS：****业务定制：** 直接对接报表（BI）、大屏或 API。数据已经完全加工好，通常不再进行复杂的 Join 操作

**为什么要分层：**

- [ ] **清晰的数据血缘：**分层让数据流向有迹可循。当一个报表数据错了，你可以快速定位是 ADS 的口径错了，还是 DWD 的清洗逻辑有问题
- [ ] **减少重复开发：**如果没有 DWS（汇总层），每个业务报表都要从 ODS 原始表开始算。有了公用的汇总层，大家直接取数，开发效率倍增
- [ ] **隔离原始数据异常：**业务系统（OLTP）经常会改字段、改逻辑。通过 ODS 和 DWD 的屏蔽，可以确保下游的报表不会因为业务数据库的一次微调而全量崩溃。
- [ ] **复杂问题简单化：** 将复杂的业务逻辑拆解成多个层级的简单逻辑。每一层只解决一个问题（比如 DWD 只管清洗，DWS 只管指标），降低了代码维护难度。



### DWD 层为什么要维度退化，如何进行

- [ ] **性能优化：** 维度退化可以减少查询时的 Join 操作
- [ ] **易用性：** 将常用的维度属性直接冗余到事实表中，让下游 DWS 层取数更简单
- [ ] **常用维度冗余：** 将用户等级、商品类目、店铺名称等高频使用的属性，直接写入事实表。
- [ ] **剥离缓慢变化维 (SCD)：** 对于经常变动的属性，需要权衡。通常在 DWD 记录的是**发生时刻的快照属性**（例如：下单时的会员等级，而不是当前的会员等级）。
- [ ] **保留维度 ID：** 即使进行了退化，也要保留关联维度表的原始 ID，以便特殊情况下回溯查取更多冷门属性。





### DWD 层如何进行事实表设计

- [ ] **选择业务过程：** 确定你要建模的业务动作（如：下单、支付、退款）。通常一个 DWD 表对应一个业务过程
- [ ] **确定粒度：** **这是最关键的一步。** 必须明确一行数据代表什么（如：子订单订单项、单次点击）。**面试加分项：强调“尽量选择最小粒度”，**以支持未来多维度的灵活汇总。
- [ ] **确认维度：** 确定与该业务过程相关的环境信息（如：时间、用户、商品、店铺、推广渠道）。
- [ ] **确定事实：** 确定可累加的数字度量（如：GMV、折扣金额、下单次数）。
- [ ] **事实表分类：** **事务事实表：**记录单次业务事件；**周期快照事实表：**记录存量信息（如：每日库存余量、账户余额）；**累积快照事实表：**用于跟踪工单、物流等**多阶段状态**。一行数据记录一个生命周期的所有关键时间点



### 实时数仓里你是如何进行维度建模的？

**以电商订单下单过程为例：**

1. ##### 选取业务过程

- **核心活动**：确认下单（Order Checkout）。
- **关注点**：关注订单从创建到付款的业务流，确保数据一致性。

2. ##### 定义粒度（原子级）

- **定义**：**订单中的每个SKU明细（子项）**。
- **示例**：若一个订单包含2件衣服和1个水杯，事实表中记录为2行数据。
- **目的**：提供最高灵活性，支持从单品到总单的任意维度汇总。

##### 3. 确认维度（环境描述）

- **时间维度**：下单日期、时间点、是否大促期间。
- **商品维度**：SKU ID、商品名称、品牌、类目、颜色规格。
- **用户维度**：买家ID、性别、会员等级、注册城市。
- **店铺维度**：卖家ID、店铺名、经营类目、评分。

##### 4. 确认事实（数字度量）

- **数量类**：销售件数。
- **金额类**：商品原价、优惠券抵扣金额、运费、最终实付金额。

##### 5. 核心结构

- **事实表**：存放“订单ID + 各维度ID（外键）+ 度量值（数字）”
- **维度表**：存放“各维度ID + 详细文字描述属性”





### 事实表设计分几种，每一种都是如何在业务中使用

- [ ] **事务事实表：**以订单事务事实表为例，每条数据是一次下单事务；粒度最细，数据一旦插入不可修改，不可删除，只记录那一刻发生了什么
- [ ] **周期快照事实表：**以每日库存事实表为例，记录某一天某商品的库存量，如：2025-10-01，商品甲，期末库存 100 件；
- [ ] **累积快照事实表：**以下单事实表为例，记录下单生命周期的进度，包括单、支付、发货、收货等多个时间字段；而且是动态更新，当动作发生时，再把对应的时间填上



### 一致性维度

- [ ] 在不同的事实表中共享的、定义完全相同的维度。无论数据流向哪个业务域，对同一实体的描述（ID、名称、属性）必须是同步且唯一的。
- [ ] **实现**：通常将这些维度维护在 HBase、Redis 或外部维表中，供所有实时计算任务统一关联。
- [ ] **应用场景**：在“实时订单流”和“实时购物车流”中，如果使用的“商品类目”定义不一致（一个叫“电子产品”，一个叫“3C数码”），后续的跨域分析就会产生歧义

### 一致性事实

- [ ] 在多个事实表中出现的、**度量单位**和**业务口径**完全相同的指标。
- [ ] **电商例子：**在“实时销售大屏事实表”中，GMV 是下单即计算；在“实时商家结算事实表”中，GMV 也必须是下单即计算。如果大屏算的是“含运费金额”，而结算算的是“不含运费金额”，但两者都叫“GMV”，就会导致管理层和业务方的数据对不上。
- [ ] 一致性事实要求两者的计算逻辑、数据类型、单位（如：均为人民币，保留两位小数）完全统一。

### DIM层如何通过 Flink CDC 把 MySQL 的维度数据实时同步到 HBase，并保持一致性？

**架构：**

- [ ] **Source层**：Flink CDC 充当 MySQL 的从库，读取全量数据后，通过 Binlog 持续监听增量变更
- [ ] **转换层：**Flink 将 Binlog 里的数据行转为 HBase 的 Put 或 Delete 对象
- [ ] **Sink层**：使用 Flink 的 HBase Connector 将数据写入指定的 RowKey 对应列族

**保持“一致性”：**

- [ ] **状态一致性 ：**Flink 开启 **Checkpoint** ，Flink CDC 记录了 Binlog 的读取偏移量；如果 Flink 任务挂了，重启后会**从上一个成功的 Checkpoint 恢复**，确保不丢数据；由于 **HBase 的写入是幂等**的（相同 RowKey 的 Put 操作会覆盖旧值），通过 Checkpoint 机制可以实现 **Exactly-Once（精准一次）** 的逻辑一致性
- [ ] **顺序一致性：**MySQL 是并发修改的，如果 Binlog 乱序到达 HBase，旧数据可能会覆盖新数据；**解决方案**：Flink CDC 默认保证了单并行度下的 Binlog 事件顺序。HBase 自带 **Timestamp（时间戳）**。在写入时，可以将 MySQL 记录里的 `update_time` 作为 HBase 的版本时间戳，这样**即使数据乱序到达，HBase 也会根据时间戳保留最新的值。**
- [ ]  **结构一致性：**当 MySQL 维度表增加字段时，只需在 Flink 的转换逻辑中增加对新字段的映射即可
- [ ] **删改一致性：** **更新：**Flink CDC 会捕获 UPDATE 事件，直接下发 Put 到 HBase 覆盖旧值；**删除：**必须**配置 Flink 让其捕获 DELETE** 事件，并向 HBase 发送 Delete 请求，否则 HBase 里会残留“脏数据”



### 为什么选doris而不是clickhouse储存DWS/ADS层的数据？

- [ ] **运维/开发角度：**doris兼容mysql, clickhouse是类SQL
- [ ] **部署和架构：**Doris 只有 Frontend (FE) 和 Backend (BE) 两个进程，**支持副本自动平衡**；ClickHouse依赖Zookeeper 进行副本管理和分布式协同，需要**手动Rebalance** 
- [ ] **JOIN性能：**在多表关联（星型/雪花模型）时表现非常出色，ClickHouse在处理大表 Join 时，ClickHouse 经常会出现 OOM
- [ ] **并发能力：** Doris并发处理能力较强，能够支撑上千人同时在线查询报表，ClickHouse 单次查询吞吐极高，但并发能力弱



### 事实表关联 HBase 维度表， 为什么要用Lookup Join 而不是其他类型的Join?

- [ ] **Lookup Join：**当事实流（比如评论流、支付流）进来一条数据时，Flink **主动去外部存储**（HBase）**按 Key 点查**一次对应的维度数据，而**不是把维度表加载到 Flink 状态**里, 避免Flink状态过大OOM
- [ ] **Regular Join：**会把**两侧数据全部存在 Flink State 中**，维度表（例如用户表，商品表）往往数亿行，会导致 Checkpoint 极慢甚至 OOM
- [ ] **Window Join** ：要求两条流的数据落在同一个时间窗口内，但维度数据是持久存在的，评论数据是瞬时发生的，生命周期完全错位
- [ ] **Interval Join：**  适合"**订单关联支付**"这种两个事件都**有明确时间差**的场景，维表没有这种严格的时间约束

### 异步I/O

- [ ] Flink 提供了 AsyncDataStream API，内部维护了一个未完成请求队列；每条输入数据进来后，Flink 创建一个 ResultFuture 对象放入队列，然后调用用户定义的 AsyncFunction.asyncInvoke() 方法发出异步请求；当外部系统返回结果时，通过回调把结果填入对应的 ResultFuture，Flink 再从队列中取出已完成的结果向下游发送。
- [ ] **timeout（超时时间）**：单个异步请求的最大等待时间；超时后 Flink **默认会抛异常导致作业失败重启**；如果希望超时后跳过或用默认值填充而不是重启，需要重写 `timeout()` 方法



### 异步I/O使用unorderedWait，会不会使waterMark和数据乱序，导致事件时间语义不一致？

- [ ] 在开启 Checkpoint 的情况下，即使使用 unorderedWait，Flink 也会保证Watermark 不被乱序超越；

- [ ] 也就是说，开启checkpoint后，两个watermark之间的数据可以是乱序，但乱序数据不会越过这两个watermark

- [ ] 举例：

  - 【WM=3】 数据A 数据B 数据C 【WM=5】 数据D 数据E 【WM=8】
  - 两个WM之间（比如 A、B、C 之间）顺序随便换，没关系
  - 但 A、B、C 不会跑到 【WM=5】 后面去，D、E 也不会跑到 【WM=5】 前面来
  - 总结：Watermark 就像一道道**栅栏**，数据在栅栏内可以自由乱序，但不能翻越栅栏。这样下游窗口看到某个 Watermark 时，就能安全地认为"该到的数据都到了"，事件时间语义就是一致和正确的

  

  

### 异步 I/O 与 Checkpoint 的交互

- [ ] 异步 I/O 的未完成请求队列中可能有很多"飞行中"的请求，当 Checkpoint Barrier 到来时，Flink 需要把这些**未完成的请求也纳入快照**。
- [ ] 具体行为是：Barrier 到达异步算子时，算子会等待队列中所有**在 Barrier 之前进入的请求**完成（或者将它们的状态保存下来），然后才完成 Checkpoint。
- [ ] 如果大量请求飞行中，Checkpoint 的完成时间会被拉长。所以 capacity 的调优需要同时考虑吞吐量和 Checkpoint 性能的平衡



### unorderedWait 下的 Checkpoint，如何实现精准一次？

 **Checkpoint 在异步算子中做了什么:**

- [ ] 当 Barrier 流入异步算子时，Flink 需要对**未完成请求队列做快照**
- [ ] 具体来说，它会把队列中**所有尚未 complete 的输入元素**保存到状态后端。注意保存的是**原始输入数据**，而不是异步请求本身或返回结果
- [ ] 恢复时，Flink 从 Checkpoint 中取出这些输入元素，重新发起异步请求，也就是**重放输入、重新查询**



**为什么 unorderedWait 也能正确配合：**

- [ ] **unorderedWait 本身就声明了"我不保证顺序"**。也就是说，选择 unorderedWait 的前提是下游语义不依赖顺序；既然正常运行时顺序就是不确定的，那恢复后顺序变了也不会产生新的问题
- [ ] **Barrier 对齐保证了跨算子的一致性切面：**  Barrier 把数据流切成了"Barrier 之前"和"Barrier 之后"两部分，在异步算子内部，Checkpoint 快照记录的是"Barrier 到达时，哪些输入已经进了队列但还没输出"；**恢复后这些元素会被重新处理，而 Barrier 之前已经输出给下游的元素不会被重复处理**



**例子：**

- [ ] 假设输入顺序是 A、B、C，Barrier 在 C 之后到达
- [ ] **正常运行时**，unorderedWait 可能输出 B、A、C（因为 B 的查询先返回了）
- [ ] **故障发生在 A 已输出、B 和 C 还在飞行中**。Checkpoint 快照中记录了 B 和 C 的原始输入。恢复后重新对 B 和 C 调用异步查询，这次可能输出 C、B
- [ ] 最终效果：第一次运行输出 B、A、C；故障恢复后输出 A（故障前已输出）、C、B（恢复后重新处理）。顺序不同，但**每个元素恰好被处理了一次**，这就是 Exactly-Once 的语义
- [ ] **每条数据的副作用恰好生效一次"，而不是"输出顺序完全确定**



**orderedWait 多了什么：**

- [ ] orderedWait 在上述机制之上，额外维护了一个**输出顺序缓冲区**
- [ ] 即使 B 先返回，也要等 A 完成后才按 A、B 顺序输出。Checkpoint 时同样保存未完成的输入元素，恢复后重放并重新排序
- [ ] 所以 orderedWait 提供的是**确定性输出顺序**这个额外保证，而不是"比 unorderedWait 更正确"——**两者在 Exactly-Once 语义上是等价的**
